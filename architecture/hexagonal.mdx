---
title: "Hexagonal Architecture"
description: "How the inventory service implements Ports & Adapters to isolate domain logic from infrastructure."
---

The **inventory-service** was refactored from a traditional layered architecture to **Hexagonal Architecture** (also known as Ports & Adapters). The goal is to keep the domain model and business rules free of framework dependencies — no JPA annotations, no Spring interfaces, no messaging primitives in the core.

The result: swapping H2 for MongoDB, or replacing the HTTP adapter with a gRPC adapter, requires writing a new adapter class only. The domain and use-case code are untouched.

## Before vs. after

**Before — traditional layered structure:**

```text
inventory-service/src/main/java/com/uade/inventory/
├── InventoryServiceApplication.java
├── config/
│   ├── DataInitializer.java
│   └── SecurityConfig.java
├── controller/
│   └── InventoryController.java        ← accesses Repository directly
├── entity/
│   └── Product.java                    ← domain model coupled to JPA (@Entity)
└── repository/
    └── ProductRepository.java          ← Spring Data JPA
```

Problems with this structure:
- The controller accessed the JPA repository directly — no business layer.
- `Product.java` was annotated with `@Entity`, `@Table`, `@Id` — domain coupled to JPA.
- Changing the database would require modifying the domain and the controller.
- No clear boundary between business logic and infrastructure.

**After — hexagonal structure:**

```text
inventory-service/src/main/java/com/uade/inventory/
├── InventoryServiceApplication.java
│
├── domain/                              ← CORE (no framework dependencies)
│   ├── model/
│   │   └── Product.java                ← pure POJO, no JPA annotations
│   ├── event/
│   │   └── ProductCreatedEvent.java    ← domain event
│   └── port/
│       ├── in/
│       │   └── ProductUseCase.java      ← input port (interface)
│       └── out/
│           ├── ProductRepositoryPort.java  ← output port (interface)
│           └── EventPublisherPort.java     ← output port (interface)
│
├── application/                         ← USE CASES
│   └── service/
│       └── ProductService.java          ← implements ProductUseCase
│
└── infrastructure/                      ← ADAPTERS (frameworks, DB, HTTP)
    ├── adapter/
    │   ├── in/web/
    │   │   └── InventoryController.java ← input adapter (REST)
    │   └── out/persistence/
    │       ├── ProductJpaEntity.java     ← JPA entity (infrastructure only)
    │       ├── ProductJpaRepository.java ← Spring Data JPA
    │       ├── ProductPersistenceAdapter.java ← implements ProductRepositoryPort
    │       └── ProductMapper.java        ← maps domain ↔ JPA
    └── config/
        ├── SecurityConfig.java
        └── DataInitializer.java
```

## Layer breakdown

### Domain layer

Contains the core model and port interfaces. No Spring, no JPA, no messaging. Pure Java.

### Application layer

Contains use case implementations (`@Service` classes). Depends only on the domain — never on infrastructure. Calls output ports to persist data or publish events.

### Infrastructure layer

Contains all framework-specific code: REST controllers, JPA entities, Spring Data repositories, messaging adapters, and configuration. Implements the port interfaces defined in the domain.

## Code walkthrough

### Domain model — `Product.java`

A plain Java object with no framework annotations:

```java title="domain/model/Product.java"
package com.uade.inventory.domain.model;

import java.util.Objects;

public class Product {

    private Long id;
    private String name;
    private Integer quantity;
    private Double price;

    public Product() {
    }

    public Product(String name, Integer quantity, Double price) {
        this.name = name;
        this.quantity = quantity;
        this.price = price;
    }

    public Product(Long id, String name, Integer quantity, Double price) {
        this.id = id;
        this.name = name;
        this.quantity = quantity;
        this.price = price;
    }

    public Long getId() {
        return id;
    }

    public String getName() {
        return name;
    }

    public Integer getQuantity() {
        return quantity;
    }

    public Double getPrice() {
        return price;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        Product product = (Product) o;
        return Objects.equals(id, product.id);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id);
    }
}
```

### Input port — `ProductUseCase.java`

Defines the operations the domain exposes to the outside world. The controller depends on this interface, not on any implementation:

```java title="domain/port/in/ProductUseCase.java"
package com.uade.inventory.domain.port.in;

import com.uade.inventory.domain.model.Product;

import java.util.List;
import java.util.Optional;

public interface ProductUseCase {

    List<Product> getAllProducts();

    Optional<Product> getProductById(Long id);

    Product createProduct(Product product);
}
```

### Output port — `ProductRepositoryPort.java`

Defines what the domain needs from persistence. The domain does not know about JPA or H2:

```java title="domain/port/out/ProductRepositoryPort.java"
package com.uade.inventory.domain.port.out;

import com.uade.inventory.domain.model.Product;

import java.util.List;
import java.util.Optional;

public interface ProductRepositoryPort {

    List<Product> findAll();

    Optional<Product> findById(Long id);

    Product save(Product product);

    long count();
}
```

### Use case implementation — `ProductService.java`

Implements `ProductUseCase`. Receives both output ports by constructor injection — it knows nothing about JPA or messaging specifics:

```java title="application/service/ProductService.java"
package com.uade.inventory.application.service;

import com.uade.inventory.domain.event.ProductCreatedEvent;
import com.uade.inventory.domain.model.Product;
import com.uade.inventory.domain.port.in.ProductUseCase;
import com.uade.inventory.domain.port.out.EventPublisherPort;
import com.uade.inventory.domain.port.out.ProductRepositoryPort;
import org.springframework.stereotype.Service;

import java.util.List;
import java.util.Optional;

@Service
public class ProductService implements ProductUseCase {

    private final ProductRepositoryPort repositoryPort;
    private final EventPublisherPort eventPublisherPort;

    public ProductService(ProductRepositoryPort repositoryPort, EventPublisherPort eventPublisherPort) {
        this.repositoryPort = repositoryPort;
        this.eventPublisherPort = eventPublisherPort;
    }

    @Override
    public List<Product> getAllProducts() {
        return repositoryPort.findAll();
    }

    @Override
    public Optional<Product> getProductById(Long id) {
        return repositoryPort.findById(id);
    }

    @Override
    public Product createProduct(Product product) {
        Product saved = repositoryPort.save(product);
        eventPublisherPort.publishProductCreated(
                new ProductCreatedEvent(saved.getId(), saved.getName(), saved.getQuantity(), saved.getPrice())
        );
        return saved;
    }
}
```

### Input adapter — `InventoryController.java`

Receives HTTP requests and delegates to `ProductUseCase`. No persistence or messaging code here:

```java title="infrastructure/adapter/in/web/InventoryController.java"
package com.uade.inventory.infrastructure.adapter.in.web;

import com.uade.inventory.domain.model.Product;
import com.uade.inventory.domain.port.in.ProductUseCase;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

import java.util.List;

@RestController
@RequestMapping("/api/inventory")
public class InventoryController {

    private final ProductUseCase productUseCase;

    public InventoryController(ProductUseCase productUseCase) {
        this.productUseCase = productUseCase;
    }

    @GetMapping("/products")
    public ResponseEntity<List<Product>> getAllProducts() {
        return ResponseEntity.ok(productUseCase.getAllProducts());
    }

    @GetMapping("/products/{id}")
    public ResponseEntity<Product> getProduct(@PathVariable Long id) {
        return productUseCase.getProductById(id)
                .map(ResponseEntity::ok)
                .orElse(ResponseEntity.notFound().build());
    }

    @PostMapping("/products")
    public ResponseEntity<Product> createProduct(@RequestBody Product product) {
        return ResponseEntity.ok(productUseCase.createProduct(product));
    }
}
```

### Output adapter — `ProductPersistenceAdapter.java`

Implements `ProductRepositoryPort` using Spring Data JPA and a mapper. The domain never sees `ProductJpaEntity`:

```java title="infrastructure/adapter/out/persistence/ProductPersistenceAdapter.java"
package com.uade.inventory.infrastructure.adapter.out.persistence;

import com.uade.inventory.domain.model.Product;
import com.uade.inventory.domain.port.out.ProductRepositoryPort;
import org.springframework.stereotype.Component;

import java.util.List;
import java.util.Optional;

@Component
public class ProductPersistenceAdapter implements ProductRepositoryPort {

    private final ProductJpaRepository jpaRepository;
    private final ProductMapper mapper;

    public ProductPersistenceAdapter(ProductJpaRepository jpaRepository, ProductMapper mapper) {
        this.jpaRepository = jpaRepository;
        this.mapper = mapper;
    }

    @Override
    public List<Product> findAll() {
        return jpaRepository.findAll().stream()
                .map(mapper::toDomain)
                .toList();
    }

    @Override
    public Optional<Product> findById(Long id) {
        return jpaRepository.findById(id)
                .map(mapper::toDomain);
    }

    @Override
    public Product save(Product product) {
        ProductJpaEntity saved = jpaRepository.save(mapper.toJpaEntity(product));
        return mapper.toDomain(saved);
    }

    @Override
    public long count() {
        return jpaRepository.count();
    }
}
```

## Hexagonal concepts at a glance

| Concept | Class | Role |
|---------|-------|------|
| Domain pure | `Product.java` | POJO with no framework annotations |
| Input port | `ProductUseCase.java` | Interface defining what the domain exposes |
| Output port | `ProductRepositoryPort.java` | Interface defining what the domain needs from persistence |
| Output port | `EventPublisherPort.java` | Interface defining what the domain needs from messaging |
| Input adapter | `InventoryController.java` | Receives HTTP, delegates to the use case |
| Output adapter | `ProductPersistenceAdapter.java` | Implements persistence port using JPA |
| Mapper | `ProductMapper.java` | Translates between `Product` (domain) and `ProductJpaEntity` (JPA) |

## Benefit: infrastructure is replaceable

To replace H2/JPA with MongoDB:

1. Create `ProductMongoAdapter` implementing `ProductRepositoryPort`.
2. Register it as a Spring `@Component`.
3. The domain model, use case, and controller are unchanged.

<Note>
  **Dependency inversion in practice.** The domain layer defines `ProductRepositoryPort` and `EventPublisherPort`. The infrastructure layer implements them. This means the domain dictates the contract — infrastructure adapts to the domain, not the other way around.
</Note>
